# 验证计划

CCFTS 当前最重要的短板是：公开仓库里还缺少足够多的真实脱敏案例和可复现实验记录。

本文件用于说明后续怎么验证它，而不是提前宣称它已经被充分验证。

## 当前验证状态

当前已有：

- 96 个 skill 的结构校验。
- 3 个脱敏教学 demo。
- demo 输入和 expected 输出的基本一致性检查。
- MCP 加载检查。

当前不足：

- 脱敏 demo 主要用于教学和流程演示，不足以证明真实业务效果。
- 缺少多企业、多项目、多期间样本。
- 缺少专家逐条复核记录。
- 缺少量化指标，例如错误率下降、复核效率提升、风险识别命中率等。

## 验证分级

建议把案例分为四级：

| 级别 | 名称 | 含义 |
| --- | --- | --- |
| L1 | 教学样例 | 人工构造或简化脱敏样例，只用于说明流程 |
| L2 | 实务脱敏样例 | 来自真实业务结构，已脱敏，可公开讨论 |
| L3 | 人工复核样例 | 有财务、税务、审计或业务专家复核记录 |
| L4 | 可复现实验 | 输入、输出、复核、修正、指标都可重复验证 |

当前公开 demo 主要属于 L1。

## 每个案例应记录什么

建议每个验证案例至少记录：

- 场景名称。
- 主体类型：项目部、项目公司、分公司、子公司、集团等。
- 资料期间。
- 输入资料清单。
- 已做脱敏处理说明。
- 使用了哪些 skill 或场景包。
- AI 输出摘要。
- 人工复核结论。
- 发现的问题。
- 修正后的规则或文档。
- 是否允许公开为 demo。

## 建议验证指标

不同场景可以使用不同指标：

| 场景 | 可观察指标 |
| --- | --- |
| 财务快报资料检查 | 主体判断是否正确、科目映射错误数、重大差异识别数、缺失资料提示是否完整 |
| 增值税跨区域预缴 | 跨区域判断是否正确、计税方式判断是否正确、分包扣除和地方口径是否提示 |
| 清收清欠 | 欠款分层是否合理、责任部门是否清楚、下一步动作是否可执行 |
| 质保金 | 到期判断是否正确、合同条款是否引用准确、回收风险是否提示 |
| 项目经营分析 | 成本偏差识别、现金流风险、亏损预警、管理动作建议是否有用 |

## 验证流程

建议按这个流程做：

1. 准备脱敏资料。
2. 填写项目背景包。
3. 选择相关 skill 或场景包。
4. 让 AI 输出结果。
5. 用人工复核模板逐项检查。
6. 记录错误、遗漏和有用输出。
7. 修正 skill、场景包或说明文档。
8. 如可公开，整理成 demo 或 evidence log。

## 不应做的验证

不要用以下方式证明项目有效：

- 只展示一段看起来漂亮的 AI 回答。
- 不提供输入资料和复核过程。
- 不说明脱敏方式。
- 不区分教学样例和真实脱敏样例。
- 不记录错误和修正。

## 当前优先征集的验证案例

最需要补充：

- 清收清欠。
- 资金计划。
- 竣工结算。
- 质保金到期回收。
- 项目经营分析。
- 亏损项目预警。
- 增值税跨区域预缴。
- 财务快报资料检查。
