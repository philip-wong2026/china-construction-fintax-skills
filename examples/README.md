# 脱敏演示场景

> 以下所有数据均为脱敏教学样例，不代表真实企业报表，不构成财税、审计或合规意见。

---

## 五个演示场景一览

| 脱敏演示 | 模拟什么 | 适合谁 | 主要管理价值 |
|---------|---------|--------|------------|
| 项目部快报资料检查 | 项目部上报月度快报前，总部或上级单位希望先检查资料是否齐全、口径是否一致、异常指标是否需要解释 | 项目财务、总部财务、经营分析人员 | 减少资料反复退回，让项目先补资料，帮总部形成问题清单 |
| 项目公司经营口径识别 | 模拟投资类项目公司（独立法人）与普通施工项目部（非法人）在报表、资金、资产负债、利润口径上的差异 | 投资项目财务、项目公司财务、总部管理人员 | 防止把项目公司当普通项目部分析，防止把法人主体风险和项目现场风险混在一起 |
| 增值税跨区域预缴判断 | 模拟施工项目跨地区经营时，税务人员或项目财务需要先判断是否涉及增值税跨区域预缴、资料是否齐全、哪些事项需要税务岗复核 | 税务岗、项目财务、共享中心 | 帮助整理判断路径，防止只看税率，防止忽略项目所在地、机构所在地、计税方式、发票和预缴记录 |
| 清收清欠台账分析与催收优先级 | 模拟清收清欠人员拿到一份应收账款台账后，AI 按账龄分层、业主类型、确权状态、资料完备性排序催收优先级，识别高风险项目 | 清收清欠岗、项目财务、商务部、法务部 | 防止只看金额大小不看账龄，防止对"该走法律程序的"还在发函、对"就差最后一步的"反而没人跟 |
| 往来款分层清收管理样例 | 模拟某建设集团拿到脱敏应收账款台账后，AI 按账龄、确权状态、开票回款差额和责任部门分层管理 | 清收清欠岗、项目财务、商务合约人员、项目经理部 | 帮助识别先催收、先确权、先补资料、先处理争议的不同路径，避免直接给坏账判断 |

## Demo 1：项目部快报资料检查

**目录**：`demo-project-unit-flash-report/`

**模拟什么场景**：
项目部每月要向上级单位上报财务快报。在上报前，总部或财务共享中心希望先检查：资料是不是齐全、科目口径对不对、取整规则对不对、有没有异常指标需要解释。

**为什么这个场景适合用 CCFTS**：
快报检查有很多固定规则（资料格式、科目映射、取整规则、平衡验证），这些规则写下来就是一套检查清单。AI 可以先按清单逐项检查，把明显的问题找出来，人再复核。这样可以减少反复退回、减少遗漏。

**你需要提供什么**：
- 脱敏科目余额表（标准列位格式）
- 报表期间
- 说明主体是项目部/总包部（不是独立法人）
- 如果是年末最后一个月，要说明

**AI 应该先检查什么**：
1. 资料是否完整（有没有缺少关键科目）
2. 主体判断是否正确（是不是项目部而不是法人公司）
3. 取整规则是否匹配（项目部用截尾取整，不是四舍五入）
4. 平衡关系是否成立（项目部权益接近零，资产应该约等于负债）

**AI 可以输出什么**：
- 资料缺口清单
- 科目映射建议
- 利润表和资产负债表草稿
- 异常指标和差异说明
- 必须人工复核的事项

**哪些地方必须人工复核**：
- 主体层级是否与实际组织架构一致
- 非季度末是否有未结转的福利费和管理费
- 异常差异是否经得起业务解释
- 其他应付款的计算方法是否正确

**这个 demo 的局限**：
- 只覆盖了基本的报表科目，未覆盖现金流量表、业务板块表
- 演示数据为简化版本，实际项目的科目种类更多
- 未考虑企业个性化报表格式

**不代表什么**：
- 不替代财务负责人审批
- 不构成正式上报前的合规意见
- 不保证与企业现有报表系统完全一致

## Demo 2：项目公司经营口径识别

**目录**：`demo-spv-flash-report/`

**模拟什么场景**：
投资类项目、项目公司（为项目设立的独立法人公司）与普通施工项目部的财务口径完全不同——项目公司有注册资本、有银行贷款、要独立纳税、有完整的所有者权益。如果财务分析时把项目公司当普通项目部处理，会得出完全错误的结论。

**为什么这个场景适合用 CCFTS**：
项目公司的财务口径判断涉及一系列规则：有没有施工科目、所有者权益是不是正常、其他应付款怎么算、利润表和资产负债表用什么取整规则。这些规则如果不写清楚，每次都要重新解释。AI 可以先按规则判断主体类型，再提醒分析人员注意口径差异。

**你需要提供什么**：
- 脱敏科目余额表
- 基本背景说明（是否为独立法人、是否涉及投资管理）
- 报表期间

**AI 应该先检查什么**：
1. 有没有施工类科目（合同结算、合同履约成本）——有就是施工企业，没有可能是投资管理类
2. 所有者权益是不是完整——完整的是独立法人，接近零的是内部项目部
3. 有没有内部往来科目——这是区分法人主体和内部单位的重要信号

**AI 可以输出什么**：
- 主体类型判断和判断依据
- 与项目部的关键口径差异
- 利润表、资产负债表编制提示
- 必须人工复核的事项

**哪些地方必须人工复核**：
- 主体类型是否符合工商登记和内部管理口径
- 是否涉及合并报表（如该法人公司下面还有子公司）
- 银行贷款、对外担保等法人特有事项是否已考虑
- 其他应付款的计算方法是否选择了正确的口径

**这个 demo 的局限**：
- 项目公司的实际情况比较复杂（可能涉及多种业务类型、多个项目）
- 演示数据为简化版本，未包含 PPP、BOT 等复杂模式
- 未覆盖项目公司特有的税务申报差异

**不代表什么**：
- 不构成对项目公司法律主体地位的判断
- 不替代对项目公司实际经营情况的分析
- 不能用于正式审计或监管报送

## Demo 3：增值税跨区域预缴判断

**目录**：`demo-vat-prepayment/`

**模拟什么场景**：
施工项目跨地区经营时，项目财务或税务人员需要判断：这个项目要不要在项目所在地预缴增值税？适用一般计税还是简易计税？预缴多少？还要缴哪些附加税费？资料是否齐全？哪些地方需要税务岗复核？

**为什么这个场景适合用 CCFTS**：
增值税跨区域预缴涉及多个判断条件（项目地址、机构所在地、合同类型、计税方式、分包扣除、附加税费），判断路径很长，容易遗漏关键条件。AI 可以先按判断路径梳理，整理出"你需要确认什么""你还缺什么资料""哪些地方必须问税务局"。

**你需要提供什么**：
- 项目所在地和机构所在地
- 合同类型（纯施工、清包工、甲供、EPC 等）
- 合同金额和分包金额
- 是否已开具发票、是否已收款

**AI 应该先检查什么**：
1. 项目是否跨县/市/区——如果都在同一个地方，可能不需要跨区域预缴
2. 是一般纳税人还是小规模纳税人——决定了预征率和能否抵扣
3. 是否清包工或甲供工程——决定能否用简易计税（注意甲供工程 2026 年起不能再用简易计税）
4. 分包款有没有合规发票——没有合规发票不能扣除

**AI 可以输出什么**：
- 是否需要跨区域预缴的判断
- 适用一般计税还是简易计税的判断路径
- 预缴税额计算（含附加税费）
- 资料缺口清单
- 需要咨询当地税务机关的事项

**哪些地方必须人工复核**：
- 合同事实是否支持简易计税（不能只看合同名义，要看实际业务）
- 项目所在地税务机关是否有特殊要求（各地执行可能有差异）
- 分包扣除凭证是否合规（必须取得增值税发票）
- 甲供工程的合同日期是否在 2026 年 1 月 1 日之后（之后不能选简易计税）
- 最终申报结果必须由税务岗确认

**这个 demo 的局限**：
- 演示数据只覆盖了两个项目，实际企业可能同时有几十个跨区域项目
- 未覆盖金税四期的三方比对（开票、申报、预缴）
- 未覆盖电子税务局的实际操作流程
- 各省税务机关的具体要求可能不同，demo 基于通用规则

**不代表什么**：
- 不构成正式税务申报意见
- 不能替代税务师或企业税务岗的专业判断
- 不能直接用于电子税务局填报

## Demo 4：清收清欠台账分析与催收优先级

**目录**：`demo-collection-clear-arrears/`

**模拟什么场景**：
清收清欠人员或项目财务拿到一份应收账款台账，里面有 10 个处于不同清收阶段的项目：有的账龄超过 3 年但还没完成竣工结算；有的已发律师函但业主经营困难；有的属于政府项目但财政拨款滞后；有的处于正常回款期。需要先按账龄分层、确权状态、业主类型、资料完备性排序催收优先级，再针对每个项目给出具体建议动作。

**为什么这个场景适合用 CCFTS**：
清收清欠的判断规则（什么账龄用什么手段、什么业主类型走什么路径、什么阶段升级什么动作）可以写成一套检查清单。AI 可以先按规则排序分级，找出"手段和账龄不匹配"的项目（比如逾期 3 年还在发催收函、或者还没确权就准备起诉），让清欠人员把精力花在最该花的地方。

**你需要提供什么**：
- 脱敏应收账款台账（含合同资产、质保金）
- 合同关键信息（业主、合同金额、结算条款、质保金比例）
- 已催收记录（催收函、回函、律师函、会议纪要）
- 竣工结算进度（是否已完、是否已审计审定）
- 如有预计损失计提政策，一并提供

**AI 应该先检查什么**：
1. 欠款账龄分层——按 <6 个月 / 6 个月-1 年 / 1-2 年 / 2-3 年 / >3 年分层，确认每一层的余额和占比
2. 是否已确权——未确权的不能进入法律程序，必须先推动竣工结算或审计审定
3. 是否已开票——部分业主需要先收到发票才能安排付款
4. 是否已回款——区分"累计结算−累计收款"的差额，已到期未付部分是催收重点
5. 责任部门——财务部/商务部/清欠办/法务部是否已明确
6. 催收状态——已用的手段是否与账龄匹配（逾期 > 2 年还在电话催收就是手段不够）
7. 资料缺口——缺验收单、缺审计审定单、缺竣工决算批复、缺业主回函——这些缺口阻碍了下一步动作
8. 高风险项目——民企业主（信用风险）、>3 年未确权（时效风险和确权风险）、质保金占比高（回收周期长）

**AI 可以输出什么**：
- 催收优先级清单（按紧急程度排序）
- 风险分层（黑色/红色/橙色/黄色/绿色）
- 每个项目的建议动作和升级路径
- 资料缺口汇总（哪些项目需要先补资料才能进入下一阶段）
- 必须人工复核的事项

**哪些地方必须人工复核**：
- 业主的实际经营状况和信用状况（台账可能无法反映最新变化）
- 是否涉及后续项目合作（如果该业主还有在建或后续项目，催收策略需要调整）
- 法律手段的可行性（诉讼时效、合同管辖、保全担保成本）
- 竣工结算的客观难度（结算争议的实质内容）
- 质保金是否已经到期、是否有质量问题争议
- 坏账计提是否充足、核销是否符合企业内部权限

**这个 demo 的局限**：
- 演示数据为简化版本，实际企业的应收台账可能涉及数百个项目
- 仅基于台账信息排序，未考虑业主关系、后续合作、地方政策等"非账面因素"
- 未覆盖以资抵债、债务重组、核销的具体操作流程
- 各企业的催收制度、审批权限、分级标准可能不同

**不代表什么**：
- 不构成法律建议（法律手段必须咨询专业律师）
- 不替代清收清欠人员对实际情况的判断
- 不能直接作为起诉决策依据
- 不替代坏账计提的专业判断

## Demo 5：往来款分层清收管理样例

**目录**：`demo-receivables-collection/`

**模拟什么场景**：
某建设集团拿到一张脱敏应收账款台账后，希望 AI 不只是按金额大小排序，而是按账龄区间、当前状态、开票回款差额和责任部门，先形成分层清收建议。

**为什么这个场景适合用 CCFTS**：
清收清欠不是单纯会计问题，而是管理问题。同一笔往来款，可能是已确权但回款慢，也可能是未确权导致不能直接催收，还可能是争议中需要先由商务、项目部或法务处理。AI 应先帮人分层，而不是直接判断坏账。

**你需要提供什么**：
- 脱敏应收账款台账
- 账龄区间
- 已计量、已开票、已回款金额
- 当前状态（已确权/未确权/争议中）
- 责任部门

**AI 可以输出什么**：
- 分层催收优先级
- 资料缺口清单
- 风险标记
- 建议动作
- 必须人工复核的事项

**不代表什么**：
- 不构成坏账判断
- 不构成核销建议
- 不构成诉讼建议
- 不替代清收清欠人员和法务人员复核

## 推荐使用顺序

1. 先看 Demo 3（增值税跨区域预缴），因为它最直观：输入几个条件，输出判断结果。
2. 再看 Demo 1（项目部快报检查），因为它最贴近项目财务的日常工作。
3. 再看 Demo 5（往来款分层清收管理样例），因为它更像一个管理场景，不是会计教学。
4. 再看 Demo 4（清收清欠台账分析），因为它展示了 AI 如何在复杂台账中分层排序和匹配手段。
5. 最后看 Demo 2（项目公司口径识别），因为它的场景更复杂，涉及主体类型的根本差异。

## 注意

- 所有数据均为脱敏教学样例，不是真实企业数据。
- Demo 的目的是让新用户看懂 CCFTS 的工作方式，不是证明 AI 一定正确。
- 正式业务使用前，必须换成你自己的脱敏资料，并由具备专业能力的人员复核。
